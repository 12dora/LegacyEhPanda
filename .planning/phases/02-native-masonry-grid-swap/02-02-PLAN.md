<objective>
Gate the production swap behind a feasibility spike (Success Criterion 1 / SR-1). Temporarily wire the
`MasonryLayout` candidate into the live `.thumbnail` call site, add throwaway `proposal.width` logging,
and observe the three genuinely-uncertain SwiftUI runtime behaviors on real devices before committing:
(1) reflow when Kingfisher covers settle (A2), (2) suppression of placement animation on fetch-more
append (A1/D-31), (3) scrolling parity (SR-3). Produce the column-count sign-off table and freeze `m`.

Purpose: The parity math is already proven by Plan 01's unit tests, but a live `LayoutSubviews` cannot
be synthesized in a unit test. This spike is the real go/no-go: if reflow-on-load or animation
suppression cannot be achieved with the `Layout` approach, that is the BLOCKER this gate exists to
surface — before any dependency is removed and while rollback is a single-commit revert.

Output: a candidate `MasonryLayout` rendering the live grid, the measured column-count table for owner
sign-off, and a frozen `m`. No dependency is removed and no legacy code is deleted in this plan.

Implements the SR-1 spike gate and D-23 (freeze m), D-31 (verify animation suppression), D-29/A2 (verify reflow).
</objective>
