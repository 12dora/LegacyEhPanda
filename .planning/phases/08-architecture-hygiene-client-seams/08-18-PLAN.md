<objective>
Close GAP-04 (QUAL-01, blocker): `Scripts/check-cookie-logging.sh` is name-keyed, not data-flow-aware.
It only inspects calls on a receiver literally named `logger` (line 88) and matches a fixed token
inventory inside interpolation text (lines 5–6, 43–49). Verification proved a fixture that assigned
`cookie.value` to an arbitrarily-named local and logged it at `.public` through a Logger variable NOT
named `logger` sailed through the gate — the cookie-logging privacy invariant is unenforced against
routine local renaming.

Fix: harden the gate so a cookie value reaching a log sink is rejected regardless of the alias name or
the Logger receiver variable name (a conservative reject is acceptable as long as the clean tree still
exits 0). Add an executable fixture harness with negative fixtures for the aliased-value case and the
alternate-Logger-receiver case, wired so the gate is proven to reject them, plus an accepted `.private`
case. The current source tree MUST still pass (exit 0).

Purpose: this gate is the QUAL-01 security control; a bypassable gate is no control. Make the
enforcement resistant to aliasing and Logger renaming, and prove it with executable negatives.
Output: a hardened gate that stays green on clean source and is proven — by a committed fixture
harness — to reject the two evasion classes the verifier demonstrated.
</objective>

    Strengthen `check-cookie-logging.sh` so the cookie-logging invariant holds under aliasing and
    Logger renaming. Two required behaviors:

    1. Receiver-name independence: stop keying the scan to a receiver literally named `logger`.
       Recognize any log-method call at `.debug/.info/.notice/.warning/.error/.fault/.trace` on any
       receiver (any identifier calling one of those methods), so a Logger stored under a different
       variable name is still inspected.

    2. Alias awareness (conservative is acceptable): reject a log interpolation that carries a cookie
       value even when the value was routed through an ordinary local alias. Implement this by tainting
       locals: an identifier assigned from a cookie-bearing right-hand side (an expression containing a
       token from the cookie inventory, e.g. a `cookie`/cookie-name source or a `.value` off such a
       source) becomes cookie-bearing for the remainder of its file scope, and a subsequent
       non-`.private` log interpolation referencing that identifier is a violation. A deliberately
       conservative over-approximation is acceptable PROVIDED the current clean source tree still exits
       0 — if any conservative rule would flag clean source, tighten it until the clean tree passes.

    Preserve the accepted case: a cookie value interpolated at `privacy: .private` is allowed (the
    existing `.private` exemption at lines 77–78 stays). Keep the `getCookiesDescription`
    clipboard-only consumer check.

    Parameterize the scan root: accept an optional scan-root argument (first positional argument or an
    environment variable), defaulting to `AppPackage/Sources`, so the fixture harness (Task 2) can
    point the gate at a fixtures directory without disturbing the default clean-tree scan. When a scan
    root is supplied, scope or skip the source-path-specific `getCookiesDescription` consumer check so
    that a fixture rejection is attributable to the cookie-logging rule (not a missing-consumer error).

    Do not add any SwiftLint suppression. Keep the script POSIX `sh`-compatible as today.
